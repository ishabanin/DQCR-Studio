# DQCR Studio — Пользовательская документация

**Версия:** 1.0  
**Дата:** 22 марта 2026

## 1. О чем этот документ

Этот гайд описывает работу в DQCR Studio как в ежедневном инструменте: от создания проекта до проверки и сборки SQL-артефактов.

Целевая аудитория:
- Data Engineer, который ведет DQCR-проекты;
- аналитик, которому нужен no-code/low-code доступ к настройке моделей;
- администратор, который управляет шаблонами, правилами и каталогом.

---

## 2. Основные сущности

- `Проект` — корневая папка с `project.yml`, `contexts/`, `parameters/`, `model/`.
- `Контекст` — окружение/вариант запуска (например, `default`, `vtb`).
- `Модель` — папка `model/<ModelId>/` с `model.yml` и SQL/workflow-структурой.
- `Параметры` — глобальные (`parameters/*.yml`) и модельные (`model/<ModelId>/parameters/*.yml`).
- `Workflow cache` — вычисляемый слой для Lineage/инспекторов/автодополнения.

---

## 3. Быстрый обзор интерфейса

### 3.1 Режимы

- `Hub mode` — список всех проектов (создание, импорт, поиск, фильтры, редактирование метаданных).
- `Workbench mode` — рабочая IDE выбранного проекта.

### 3.2 Основные области Workbench

- `TopBar` — выбор проекта/контекста, Validate, Build, роль, тема, multi-context.
- `Sidebar` — дерево файлов проекта + операции с файлами и моделями.
- `TabBar` — вкладки:
  - `Project info`
  - `Lineage`
  - `Model editor`
  - `SQL editor`
  - `Parameters`
  - `Validate`
  - `Build`
  - `Admin`
- `Bottom panel` — Terminal / Logs / Output.
- `Status bar` — текущее состояние проекта/контекста.

## 4. Сценарии использования

## 4.1 Сценарий: создать новый проект

1. Откройте Hub (`All projects`).
2. Нажмите `+ New project`.
3. Заполните:
   - `Project ID` (уникальный идентификатор);
   - имя/описание;
   - тип видимости (public/private);
   - теги (опционально).
4. Подтвердите создание.
5. Откройте проект кнопкой `Open` (карточка или таблица).

Результат:
- проект появляется в списке;
- создается базовая файловая структура;
- проект можно сразу открыть в Workbench.

## 4.2 Сценарий: импортировать существующий проект

1. В Hub нажмите `↓ Import`.
2. Выберите файлы/папку проекта.
3. При необходимости задайте `project_id`, имя и описание.
4. Подтвердите импорт.

Результат:
- проект регистрируется в Hub как импортированный;
- структура доступна в Sidebar без ручной донастройки.

## 4.3 Сценарий: открыть проект и навигироваться по структуре

1. Выберите проект в Hub.
2. В Workbench используйте Sidebar:
   - разворачивайте `contexts`, `parameters`, `model`;
   - открывайте SQL/YAML файлы;
   - переключайтесь между вкладками редактора.
3. Используйте контекстные действия в дереве:
   - `new-file`, `new-folder`, `new-model`;
   - `rename`, `delete`.

Результат:
- изменения файлов отражаются сразу в проекте и участвуют в дальнейшей валидации/сборке.

## 4.4 Сценарий: создать модель

Варианты:
- из `Project info` (кнопка создания модели);
- из Sidebar (`new-model` в корне/`model`).

После создания:
- появляется `model/<ModelId>/model.yml`;
- можно перейти в `Model editor` для заполнения структуры.

## 4.5 Сценарий: редактировать model.yml в Visual/YAML режиме

1. Откройте вкладку `Model editor`.
2. Выберите модель в селекторе.
3. Работайте в `Visual` режиме:
   - `target_table`;
   - атрибуты (добавление, редактирование, reorder);
   - workflow folders и materialization;
   - CTE-настройки.
4. Переключитесь в `YAML` для точечного ручного редактирования.
5. Сохраните модель.

Важно:
- включен двусторонний sync форма ↔ YAML;
- при синтаксических ошибках YAML переключение и сохранение может блокироваться до исправления.

## 4.6 Сценарий: импорт атрибутов из каталога в модель

1. Убедитесь, что каталог загружен (см. сценарий 4.12).
2. В `Model editor` откройте импорт сущности.
3. Выберите сущность каталога.
4. Выберите стратегию объединения (добавить/обновить).
5. Примените импорт и сохраните модель.

Результат:
- `target_table.attributes` обновляются;
- подсвечиваются добавленные/обновленные поля;
- изменения участвуют в автодополнении и проверках.

## 4.7 Сценарий: редактировать SQL и использовать @config-инспектор

1. Откройте SQL-файл из Sidebar.
2. Во вкладке `SQL editor`:
   - редактируйте SQL;
   - сохраняйте (`Ctrl+S`);
   - используйте вкладки файлов с индикатором несохраненных изменений (`●`).
3. Просматривайте правую панель инспектора:
   - `Priority Chain` для @config;
   - `Parameters Used`;
   - `CTE Inspector`;
   - `Generated Output` + Preview.
4. При необходимости нажмите Preview для целевого engine.

Результат:
- SQL и связанные настройки видны в одном месте;
- проще находить источник конфигурации и конфликтов.

## 4.8 Сценарий: управлять параметрами (global/model) и тестировать значения

1. Перейдите на вкладку `Parameters`.
2. Выберите scope:
   - `global`;
   - `model:<model_id>`.
3. Создайте или откройте параметр.
4. Заполните:
   - name, description, domain_type;
   - values по контекстам (`static`/`dynamic`).
5. Нажмите `Test` для проверки резолва значения.
6. Сохраните.

Результат:
- параметр доступен в SQL через шаблонные выражения;
- виден preview resolved value для выбранного контекста.

## 4.9 Сценарий: анализировать граф зависимостей (Lineage)

1. Откройте вкладку `Lineage`.
2. Выберите модель.
3. Используйте:
   - режимы `horizontal/vertical/compact`;
   - фильтрацию по контексту;
   - поиск узлов;
   - выбор узла для деталей.
4. При необходимости экспортируйте граф в PNG.
5. Если кэш устарел/с ошибкой, запустите rebuild workflow.

Результат:
- видна актуальная DAG-структура модели и зависимости между шагами.

## 4.10 Сценарий: запустить Validate и применить Quick Fix

1. Нажмите `Validate` в TopBar или откройте вкладку `Validate`.
2. Выберите категории правил (например, `general`, `sql`, `descriptions`).
3. Запустите проверку.
4. Используйте фильтры `All / Passed / Warn / Errors`.
5. Для ошибки:
   - `Go to ...` для перехода в нужный экран/файл;
   - `Quick fix` для типовых случаев (`add_field`, `rename_folder`).
6. При необходимости откройте превью diff перед применением фикса.

Результат:
- список проблем структурирован по категориям;
- часть ошибок устраняется без ручного редактирования.

## 4.11 Сценарий: собрать проект (Build), просмотреть артефакты и скачать

1. Откройте вкладку `Build`.
2. Настройте параметры сборки:
   - `engine`: `dqcr`, `airflow`, `dbt`, `oracle_plsql`;
   - `context`;
   - `model`;
   - `dry_run` (опционально);
   - `output_path` (опционально).
3. Запустите Build.
4. Следите за прогрессом (WebSocket).
5. После завершения:
   - откройте сгенерированные файлы в дереве output;
   - сравните с предыдущей сборкой (diff);
   - скачайте файл или ZIP.

Результат:
- артефакты сборки доступны в истории и для выгрузки.

## 4.12 Сценарий: загрузить каталог (Admin/Catalog)

1. Откройте `Admin` (роль должна быть `admin`).
2. Перейдите в блок каталога.
3. Загрузите `.xlsx` файл каталога.
4. Укажите label версии (опционально).
5. Дождитесь статуса `available`.

Результат:
- каталог доступен для поиска сущностей;
- Model Editor может импортировать атрибуты из каталога.

## 4.13 Сценарий: администрировать шаблоны и правила

1. Откройте `Admin`.
2. Вкладка `Templates`:
   - выберите шаблон (`flx`, `dwh_mart`, `dq_control`);
   - измените content/rules.folders;
   - сохраните.
3. Вкладка `Rules`:
   - редактируйте id/severity/pattern/enabled;
   - запустите inline-тест на SQL-примере;
   - сохраните правила.
4. Вкладка `Macros`:
   - просматривайте реестр макросов.

Результат:
- административные настройки применяются в интерфейсе и проверках.

## 4.14 Сценарий: использовать встроенный терминал

1. Разверните нижнюю панель (`Bottom panel`).
2. Выберите вкладку `Terminal`.
3. Выполняйте команды в рабочем каталоге проектов.
4. Используйте `Logs`/`Output` для просмотра служебного вывода UI-операций.

Результат:
- CLI-операции доступны, не покидая Studio.

---

## 5. Горячие клавиши

- `Ctrl/Cmd + 1..8` — переключение вкладок верхней панели.
- `Ctrl/Cmd + S` — сохранить файл в SQL Editor/редакторах.
- `Ctrl/Cmd + Space` — автодополнение в Monaco.
- `F12` — навигация к связанным объектам (где поддерживается).

---

## 6. Типовые проблемы и решения

- `Пустой список проектов`:
  - проверьте, что backend и volume `projects/` доступны;
  - обновите страницу или нажмите `Retry` в Hub.
- `Validate/Build не запускаются`:
  - проверьте выбранный проект и модель;
  - убедитесь, что backend отвечает по `/health` и `/ready`.
- `Admin недоступен`:
  - переключите роль в TopBar на `role:admin`.
- `Нет данных lineage или fallback-режим`:
  - выполните rebuild workflow для модели;
  - проверьте корректность `model.yml` и SQL-структуры.
- `Каталог не загружается`:
  - используйте только `.xlsx`;
  - размер файла до 50 MB;
  - проверьте формат входного файла каталога.

---

## 7. Практики эффективной работы

- Валидируйте изменения после каждого логического блока правок.
- Разделяйте параметры на глобальные и модельные, чтобы избежать конфликтов контекстов.
- Перед финальным Build проверяйте `Lineage` и `Priority Chain` для критичных SQL.
- Используйте теги и metadata проектов в Hub для командной навигации.
