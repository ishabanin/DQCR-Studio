# Изменения для упаковки и обновления Framework

## Кратко

В framework добавлена стандартная Python-упаковка без переноса исходников:

- добавлен `pyproject.toml`
- добавлен `setup.py`
- добавлен CLI entry point `fw2`
- сохранена текущая структура исходников `src/`
- сохранён импортный путь `FW.*`
- добавлены `__init__.py` в Python-подкаталоги framework для корректной установки пакетов
- добавено описание схемы обновления framework как внешней версии

Это сделано для того, чтобы `DQCR Studio` мог устанавливать framework как зависимость с фиксированной версией и затем легко обновлять её.

---

## Что было до изменений

Framework уже содержал рабочий код, документацию и CLI в [src/cli.py](/Users/IgorShabanin/dev/DQCR%20Studio/FTRepCBR.Workflow.FW/src/cli.py), но не имел стандартных файлов Python-пакета:

- не было `pyproject.toml`
- не было `setup.py`
- не было способа установить framework через `pip install ...`

Из-за этого интеграция в Docker и `DQCR Studio` требовала бы нестандартных решений:

- ручного `PYTHONPATH`
- симлинков
- жёсткой привязки к внутренней структуре папок

Такая схема неудобна для обновления framework, который развивается другой командой.

---

## Что изменено

### 1. Добавлен `pyproject.toml`

Файл: [pyproject.toml](/Users/IgorShabanin/dev/DQCR%20Studio/FTRepCBR.Workflow.FW/pyproject.toml)

Что задаёт:

- стандартный Python build backend через `setuptools`
- имя пакета: `ftrepcbr-workflow-fw`
- Python `>=3.10`
- runtime-зависимости:
  - `PyYAML`
  - `Jinja2`
- console script:
  - `fw2`
- версию пакета из `FW.__version__`

Почему это важно:

- framework теперь можно устанавливать стандартно
- версия становится управляемой
- Docker может ставить framework как внешнюю зависимость

---

### 2. Добавлен `setup.py`

Файл: [setup.py](/Users/IgorShabanin/dev/DQCR%20Studio/FTRepCBR.Workflow.FW/setup.py)

Этот файл сделан специально так, чтобы **не менять текущую структуру исходников**.

Сейчас код лежит в папке `src/`, но импортируется как `FW.*`.
Вместо переноса всего дерева в папку `FW/`, упаковка настраивается программно:

- пакет `FW` мапится на текущую директорию `src`
- автоматически обнаруживаются подпакеты
- автоматически включаются пакетные данные:
  - `*.yml`
  - `*.yaml`
  - `*.j2`
  - `*.html`

Почему выбран такой подход:

- он минимально вторгается в код framework
- не ломает существующие импорты
- упрощает последующие обновления от внешней команды

---

### 3. Добавлен CLI entry point `fw2`

После установки framework CLI можно запускать так:

```bash
fw2 generate <project_path> <model_name> -c default -w dqcr
```

или так:

```bash
python -m FW.cli generate <project_path> <model_name> -c default -w dqcr
```

Это важно для `DQCR Studio`, потому что backend сможет вызывать framework как обычную установленную зависимость.

---

### 4. Добавлены `__init__.py` в Python-подкаталоги

При первой реальной проверке внутри Docker выяснилось, что часть директорий framework содержала Python-код, но не была оформлена как пакет.

Из-за этого после установки `fw2` стартовал, но падал на импорте:

```python
ModuleNotFoundError: No module named 'FW.materialization'
```

Чтобы установка была стабильной, в Python-каталоги framework добавлены `__init__.py`.

Это касается, в частности:

- `materialization/`
- нескольких поддиректорий `macros/`
- workflow/macro-подкаталогов

Почему это важно:

- package installation становится полноценной
- Python-импорты работают одинаково в исходниках и после `pip install`
- Docker smoke-check проходит ближе к реальному production-сценарию

---

## Почему мы не переносили `src` в `FW`

Это было бы более инвазивное изменение.

Если framework развивается другой командой, то перенос дерева:

- усложняет синхронизацию с их репозиторием
- создаёт лишние конфликты при обновлениях
- делает наш форк тяжелее в сопровождении

Поэтому выбран более безопасный путь:

- оставить структуру как есть
- поверх неё добавить упаковку

---

## Как теперь обновлять framework

Рекомендуемая схема для `DQCR Studio`:

1. Framework хранится и развивается в отдельном репозитории.
2. Другая команда публикует версию:
   - git tag, например `v2.1.0`
   - либо commit hash
3. `DQCR Studio` устанавливает framework по зафиксированной версии.

Пример:

```bash
pip install git+ssh://git@<repo>/FTRepCBR.Workflow.FW.git@v2.1.0
```

Обновление тогда выглядит очень просто:

- было `@v2.1.0`
- стало `@v2.2.0`

После этого:

1. пересобирается backend image
2. прогоняются smoke-тесты `build/validate`
3. новая версия начинает использоваться в `DQCR Studio`

---

## Почему это удобно для обновлений

Такой подход даёт:

- явную версию framework
- воспроизводимую сборку Docker
- быстрый откат на предыдущую версию
- отсутствие ручного копирования файлов
- отсутствие завязки на локальные симлинки и нестандартный `PYTHONPATH`

Главная идея:

`DQCR Studio` не встраивает framework как произвольную папку, а использует его как внешнюю версионируемую зависимость.

---

## Как это будет использовать `DQCR Studio`

Следующий этап интеграции может выглядеть так:

1. В Docker backend устанавливается framework нужной версии.
2. Backend вызывает его через CLI:

```bash
fw2 validate <project_path> <model_name> -c default
fw2 generate <project_path> <model_name> -c default -w dqcr
fw2 build <project_path> <model_name> -c default
```

3. Backend читает:
   - generated files
   - validation reports
   - stdout/stderr framework

4. Эти результаты отдаются в API `DQCR Studio`.

---

## Что важно учесть команде framework

Чтобы модель версионирования была устойчивой, желательно:

- выпускать git tags на стабильные версии
- не ломать CLI без необходимости
- фиксировать breaking changes в changelog
- при добавлении новых runtime-зависимостей обновлять `pyproject.toml`

---

## Что важно учесть команде `DQCR Studio`

- не редактировать внутренний код framework без крайней необходимости
- интегрироваться через установленный пакет и CLI
- фиксировать конкретную версию framework в Docker
- обновлять версию осознанно и отдельно от остальных изменений backend
